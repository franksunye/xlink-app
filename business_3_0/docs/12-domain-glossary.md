# XLink 管家侧领域词汇表（地基）

> **状态**：跨版本 **规范真源**（与代码中的 TypeScript 类型、`BFF` JSON 字段应对齐）。**变更规则**：调整对象边界、字段语义或对外命名时，须在同一变更中更新本文，并在范围文档或 changelog 中留痕；避免「只改实现、不改词汇表」导致地基漂移。  
> **业界参照**见 [11-field-service-reference](./11-field-service-reference.md)（概念边界）；**本文**固定 **XLink 侧** 对象与字段含义，不做厂商对象名搬运。

### 奠基期原则：底层实体简洁优先、扩展后置

- **起步目标**：先落 **少量边界清晰的核心实体**（工单、预约、活动、客户主体等），**表与列从简**；不为「未来也许」预建扩展子系统。  
- **与「包容命名」的关系**：领域名仍可 **语义宽**（如 `WorkOrderActivity`），**首版持久化可以只有少量固定列**；「包容」指 **概念边界不窄**，**不等于**首版就上 JSON 载荷、版本号、多子表。  
- **何时谈扩展**：出现 **可量化的** 查询/约束/类型爆炸压力时，再单独立项（窄扩展表、事件流、`payload` 版本等），并在本文 **追加条款**，避免奠基阶段过度设计。  
- **不变红线**：即便极简，**预约（调度）与活动（事实流水）仍分实体/分表**，不因求简而合并（见 §1.3）。

## 1. 核心聚合（Aggregates）

### 命名分层：展示名 vs 领域实体名

- **展示名 / 业务文案**（界面、运营话术）：可偏窄、偏场景（如「跟进记录」），随产品迭代；**不作为**持久化与跨端 API 的语义真源。  
- **领域实体名**（类型、表、BFF 契约）：宜 **内聚、包容**——能覆盖该聚合在生命周期内**合理出现的各类事实**（沟通、上门、系统动作、附件元数据等），地基更稳，避免每多一种行为就新建一个并列窄实体。  
- **纪律**：新增「工单上发生的一条事实」时，优先落入 **同一地基实体的扩展**（子类型、枚举、扩展字段），而非再开一条平行时间线真源；**扩展手段（JSON、子表等）按「奠基期原则」后置**（文首）。

### 1.1 工单 `WorkOrder`

**定义**：管家侧 **任务与协作容器**，承载客户与项目语境、当前阶段下的 **行动摘要**（`context`）、**工单活动时间序**（界面可称「跟进记录」；见 §1.3）、可选订单/资料片段，以及 **零条或多条** 服务预约。  
**不等同于**：财务/合同意义上的「订单」全文；订单细节在可选 `orderInfo` 等结构中渐进补齐。

**与路由/UI 用语**：URL 使用 `/work-orders`；界面可对用户称 **「任务」**。文档与类型名统一为 **`WorkOrder`**，避免在 spec 中混用「订单」指代工单。  
**与客户地基的关系**：持久化模型宜具备 **`accountId` / `contactId`**（见 §2）；当前 Mock 中 `customer` 等字符串为 **过渡展示**，对接 CRM 后以引用关系为准。

---

### 1.2 服务预约 `ServiceAppointment`

**定义**：**单次**可调度、可执行的 **履约时间片**（上门/到场窗口），挂在工单下；同一工单可有 **多条** 预约（如初勘与正式上门）。  
**职责边界**：预约状态描述 **该时间片** 的执行/调度进展；**不替代**工单整体生命周期。工单队列维度见下文 **`group`**。

**主预约**：由 `primaryAppointment` 规则解析——优先 `isPrimary: true`，否则取 `sequence` 最小。列表卡片与工单顶层 **`appointment` / `timeText` / `assignee` / `address`** 由主预约同步，**避免双真源**；若存在 `appointments[]`，则以预约列表 + 主预约规则为真源，顶层字段为 **派生展示字段**。其中 **`assignee` 地基语义** 应对齐 **`ServiceResource`**（§2.4），字符串为过渡期。

**字段 `status`（预约）**：对接 cloud 前应收敛为 **有限枚举**（当前 Mock 为字符串占位）。**禁止**与工单 `status` 混用同一套值不加区分。

---

### 1.3 工单活动 / 时间序条目（展示名：「跟进记录」）

**领域语义**：挂在 **`WorkOrder`** 下的一条 **结构化时间序事实**（沟通、上门、勘察、系统记录、带照片的节点等），**一条一行**，为列表与审计真源；范畴 **宽于** 日常口语里的「跟进」二字。

**命名分层**：

| 层级 | 名称 | 说明 |
| --- | --- | --- |
| 展示名 | **跟进记录**（可调整） | 管家 UI 用语，偏业务沟通，**不必**与类型名一致。 |
| 实现名（当前代码） | `FollowRecord` | 历史与概念版对齐；英文 `Follow` 偏窄，**已知技术债**。 |
| 目标领域名（推荐） | `WorkOrderActivity`（或等价） | **更包容**：同一实体承载多类活动子类型（`activityType` / `source` 等），地基内聚。 |

**演进**：在 BFF / 持久化重命名时，宜 **类型与 JSON 字段一并迁移**，并在 changelog 留痕；过渡可保留 `followRecords` 键名映射期。  
**已废弃展示**：概念版扁平字符串「时间线」不在 Web 独立展示；阶段摘要进入 **本条目的正文或结构化字段**，**不**再引入与活动时间序并行的第二套字符串时间线真源。

**持久化取向（与 Task / Event「通用活动」的关系）**：

- **领域上**：`WorkOrderActivity` 与 CRM 里挂在业务父对象上的 **通用活动（Task / Event 思想）** 同构——**一行一条事实**、带时间、参与人、类型与可选扩展载荷；有利于 **表数量少、模式可控**（通常 **一张主表** + `activity_type` / `source` 枚举即可起步）。  
- **与 `ServiceAppointment` 区分**：**预约**是「可调度的时间片」真源；**活动**是「已发生/已记录的事实」真源。二者 **不合并为一张表**：预约改约、冲突检测走预约模型；沟通记录、上门打卡、系统回写走活动表——否则调度与审计会耦死。  
- **Task vs Event**：若未来与通用 CRM 对齐，可 **逻辑上**区分「待办类」与「日历时刻类」子类型；**物理上**既可 **单表**（`kind` 区分），也可 **双表**（与 Salesforce 经典 *Task* / *Event* 分拆一致）——以 **cloud 现有表结构 + 迁移成本** 选型；**领域名仍统一称 `WorkOrderActivity`**，物理映射在 Adapter 层说明即可。  
- **奠基期（首版 DB）**：优先 **单表 + 固定列 + `activity_type` 枚举** 即可；**不**把 `payload` JSON、版本化、窄扩展表当作开工前提——待确有类型或查询压力再扩展。  
- **扩展期（后置）**：若子类型差异变大，再引入 **版本化 `payload`、窄扩展表或事件** 等；仍通过 `activity_id` 关联，避免「一类型一主表」在未必要时爆炸。

---

### 1.4 项目 `Project`

**定义**：与工单并行的 **项目** 只读摘要（列表 POC）。**v0.1 不提供** `/projects/[id]` 详情路由；后续版本若开放，须在范围文档中单列。

---

### 1.5 工作台聚合 `Dashboard`

**定义**：首页所需的 **读模型聚合**（指标、入口、今日任务片段等），**不是**持久化领域根实体；字段允许随展示迭代，但 **不得**与工单/预约的持久化语义冲突（例如 `homeTasks[].orderId` 表示 **工单 id**，非 CRM 订单号——命名历史兼容，语义上应理解为 `workOrderId`）。

## 2. 客户、身份、资源与工种（地基对象）

> 与 [Field Service Core Data Model](https://developer.salesforce.com/docs/atlas.en-us.field_service_dev.meta/field_service_dev/fsl_dev_soap_core.htm) 对照：**客户主体、联系人、执行资源、派工关系** 是现场服务域的通用分层；以下为 **XLink 命名与职责**，不搬运厂商 API 名。

### 2.1 用户 `User`

**定义**：系统 **认证与授权** 主体，兼 **审计**「谁创建、谁跟进、谁改约」。  
**关系**：活动条目（`FollowRecord` / 终局 `WorkOrderActivity`）的 **`owner`**、工单/预约侧「责任人」在持久化层宜能解析到 **`userId`**（当前 POC 可为显示名，对接后须收敛）。

### 2.2 客户主体 `Account`

**定义**：**`WorkOrder` 所服务客户的稳定业务主体**（组织、门店、客户公司等——具体子类型与现网 CRM / `business` 对齐）。  
**关系**：工单应通过 **`accountId`**（或等价外键）引用 Account；界面上的客户名称、标签等为 **展示字段** 或冗余快照，**不以自由文本为长期真源**。

### 2.3 联系人 `Contact`

**定义**：挂在 **Account** 下的 **自然人**（业主、对接人、店长等）及联络角色。  
**关系**：工单 **主联系人** 宜引用 **`contactId`**；与「拨打谁的手机」强相关。`Contact` 与 `User` 可同一自然人，也可分离（业主≠登录账号）。

### 2.4 服务资源 `ServiceResource`

**定义**：可被派往现场、占用日历档期的 **师傅个人或班组**（与 FSL *Service Resource* 概念对齐；类型名便于与 cloud / 集成对话）。  
**关系**：`ServiceAppointment.assignee` 应对齐到 **`serviceResourceId`**（过渡期可并存显示名）。后续 **技能、片区、请假** 挂在此对象或其子结构，**不**堆回工单字符串。

### 2.5 预约派工 `AssignedResource`（概念名；持久化可后移）

**定义**：**一次服务预约** 与 **一个服务资源** 的分配关系（与 FSL *Assigned Resource*：联结 *Service Appointment* 与 *Service Resource* 一致）。  
**规则**：**派工以预约为锚**；同一预约多人时扩展 **主责 / 协助** 等角色。地基期允许 **每预约仅一条主分配** 的隐式实现，但语义上 **禁止**把「谁在该时段到场」仅写在工单顶层而不落到预约侧。

### 2.6 工种模板 `WorkType`（概念名；配置台可后移）

**定义**：可复用的 **服务类型模板**（标准历时、技能门槛、是否默认创建预约等；与 FSL *Work Type* 思想一致）。  
**关系**：与工单 **`taskType`**（产品展示分类）区分——终局可为 **映射**：`taskType` 面向管家话术与筛选，`workTypeId` 为模板真源；当前 Mock 仅有文案时须在对接计划中补 **映射表或外键**。

### 2.7 预留（第二梯队）

以下在 FSL 中常见；是否纳入 XLink 由 **进线方式与排程深度** 决定。引入前须在本文 **新增独立小节** 并在版本范围文档中单独立项：

| 概念（参照用语） | 典型用途 |
| --- | --- |
| `Case` | 客诉/服务单进线后转现场工单时的父级追踪。 |
| `Asset` / 服务点位 | 多房产、设备或固定点位，与单次服务地址解耦。 |
| `ServiceTerritory` | 可派工片区与地理/组织边界。 |
| `OperatingHours` / `TimeSlot` | 可服务时间窗、与 SLA 相关的营业/预约窗口。 |

## 3. 工单字段：分工与真源

| 字段 / 概念 | 含义 | 真源性质 |
| --- | --- | --- |
| `group` | **列表筛选桶**（待接单 / 待联系 / 待上门 / 待跟进），与 `FILTER_TABS` 及 `matchWorkOrderFilter` 一致。 | **稳定维度**；与 cloud 对齐时须保持枚举可映射。 |
| `status` | **工单流程或生命周期状态码**（对接后与现网码表一致，当前 Mock 为占位字符串）。 | **与后端对齐**；UI 不得自创第二套码。 |
| `statusText` | **纯展示** 短标签（紧急、跟进中等），可由码表映射生成。 | 派生展示；允许随文案优化。 |
| `taskType` | **任务类型**（产品语言：联系客户、上门服务、跟进方案等），偏「做什么类事」。 | 与现网/运营命名对齐；对接时应有映射表。 |
| `title` | 工单 **主题/摘要标题**（如具体维修项描述）。 | 业务内容。 |
| `context` | **当前阶段行动区**：主副按钮文案、说明、徽章等，驱动详情首屏行动块。 | 展示与行动聚合；深层状态仍以 `status` + 预约为准。 |
| `appointment` | 与主预约 **`windowText` 同步** 的展示字段，列表/卡片可用。 | **有 `appointments[]` 时为派生**；无预约列表时可为唯一时间真源（过渡期）。 |
| `timeText` | 详情头等处 **时间展示**，当前与 `appointment` 同步为同文案。 | **派生**；终局可合并为单一字段，合并前须改 BFF 与前端一致。 |
| `assignee` | 当前摘要上的 **责任师傅**（与主预约同步时有预约真源）。 | 有主预约时为派生；**地基语义** 对齐 `ServiceResource`（§2.4）。 |
| `address` / `distance` | 服务地址与距离展示。 | 地址可与预约覆盖字段合并规则见类型注释。 |

## 4. 预约字段（当前形状）

| 字段 | 含义 |
| --- | --- |
| `id` | 预约实体唯一标识。 |
| `status` | 预约自身状态（待联系、已预约、已完成、待签约等 **预约域** 语义）。 |
| `windowText` | **对外展示** 时间窗文案（与工单顶层 `appointment`/`timeText` 对齐的来源）。 |
| `scheduledStart` / `scheduledEnd` | 可选 ISO 时刻；用于后续排程与排序，**不替代** `windowText` 在 POC 中的展示职责。 |
| `assignee` | 该时间片责任师傅（**地基语义** 应对齐 `ServiceResource`；见 §2.4）。 |
| `sequence` | 同一工单内排序，越小越靠前。 |
| `isPrimary` | 显式主预约；与 `sequence` 二选一规则见 `primaryAppointment`。 |
| `address` | 若与工单默认地址不同则填写。 |

## 5. 命名与演进纪律（摘要）

1. **枚举先行**：`status`（工单）、`status`（预约）、`group` 对接 cloud 前，在 spec 或本文档维护 **允许取值表**；代码可用 `string`，评审以表为准。  
2. **派生字段不写第二套业务规则**：顶层 `appointment`/`timeText`/`assignee` 仅由主预约同步或过渡期单源填充。  
3. **「订单」一词**：默认指 **客户订单/商务订单** 信息块（`orderInfo`）；**不**用「订单」指代 `WorkOrder`，文档与对外 API 避免歧义。  
4. **变更同步**：修改 `business_3_0/web/lib/mock-data.ts` 中公共类型或 BFF 契约字段时，**同步更新本文**。  
5. **新增地基对象**：引入 §2 所列实体或第二梯队对象时，须先扩写本文 **§2** 或新增小节，再改代码与 API，避免「先写字段后补定义」。  
7. **奠基期从简**：首版持久化 **少表少列**；扩展模式（`payload`、子表、事件）**待需求与数据证明后再上**（见文首「奠基期原则」）。

## 6. 架构视角：可识别边界、高内聚、松耦合

> 本节回答：**领域对象是否要有「架构上可喊出来」的名字？** —— **要**。名字标识 **聚合边界**（谁在同一个一致性圆圈里一起变）；**高内聚** = 圆圈内一起演进；**松耦合** = 圆圈之间只通过 **稳定标识（id）与明确用例** 连接，不互相穿透内部细节。

### 6.1 管家现场侧（Field Ops）— 高内聚

| 领域名（架构标识） | 内聚（一起演进、理由一致） |
| --- | --- |
| `WorkOrder` | 工单生命周期、队列 `group`、与阶段相关的 `context`、对客户的业务语境；**不**承担「日历占用」的细粒度真相。 |
| `ServiceAppointment` | 单次到场的时段、预约态、与执行资源相关的 **调度真相**；多条预约同属一单时仍 **以预约为锚** 扩展。 |
| `AssignedResource` | 「哪条预约派给谁」；派工规则变，只动本边界（或预约+资源两侧接口），**不**把规则散进工单 UI 字段。 |
| `WorkOrderActivity`（目标名）/ `FollowRecord`（实现名） | 工单上 **时间序事实** 的追加与展示；新事实类型优先 **子类型/枚举扩展**（扩展实现手段见文首奠基期原则，**后置**），而非新建平行聚合。 |

### 6.2 客户与身份（CRM / Identity）— 高内聚

| 领域名 | 内聚 |
| --- | --- |
| `Account` | 客户主体、组织层级与标签策略（与现网 CRM 对齐）。 |
| `Contact` | 自然人、联络方式、角色；**不**混入登录凭据与权限。 |
| `User` | 认证、授权、审计主体；**不**承担 CRM 客户画像的全部职责。 |
| `ServiceResource` | 「谁能被派」：与 **人/班组、技能、可用性** 相关的规则；可与 `User` 关联，但 **不等于** User（业主≠账号）。 |
| `WorkType` | 工种模板、历时与技能默认值；**不**与单笔工单现场突发混写。 |

### 6.3 读模型与入口 — 刻意弱耦合

| 名称 | 角色 |
| --- | --- |
| `Dashboard` | **读模型聚合**：为首页组装指标与入口，**允许**冗余与展示专用字段；**禁止**反向写入工单/预约真源。 |
| `Project`（当前 POC） | 列表摘要；与工单 **并列展示**，通过导航弱关联，**不**在领域上合并为同一聚合。 |

### 6.4 松耦合：跨对象怎么连才「松」

1. **跨聚合默认只传 `id`**（如 `workOrderId`、`accountId`、`serviceResourceId`）；需要展示名时在 **BFF 组装 DTO** 做冗余，**不**要求对方聚合暴露内部表结构。  
2. **依赖方向**：现场执行（工单/预约/活动）**引用** CRM 与资源；CRM **不依赖** 单笔工单存在与否（避免环依赖）。  
3. **派工与日历**：以 **`ServiceAppointment` + `AssignedResource`** 为调度耦合点；`WorkOrder` 只保留 **主预约派生摘要** 供列表，不把排程引擎绑死在工单根上。  
4. **事件/集成（未来）**：跨边界状态传播优先 **领域事件或显式 API**，避免 A 聚合直接改 B 聚合的存储。

### 6.5 与「名字」的关系（小结）

- **架构上要有稳定英文名**（类型、表、OpenAPI 组件名）：对应 **§1～§2** 与上表；团队口头上可以用中文简称，但 **契约与代码以表为准**。  
- **高内聚**：改一件事只开一个 PR 的「主聚合」应清晰（例如只改预约状态机不动 Account 表）。  
- **松耦合**：两个聚合之间若每次改动都要双改多处隐式规则，说明 **边界画错或缺了显式对象**（典型补救：抽出 `AssignedResource`、抽出 `WorkOrderActivity`）。
