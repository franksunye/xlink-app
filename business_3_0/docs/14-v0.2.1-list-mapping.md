# Web v0.2.1 范围 — 工单列表 Cloud 映射（语义翻译 + Tab 归类）

> **状态**：已完成（v0.2.4，2026-05-16）  
> **日期**：2026-05-16  
> **前置**：[13-v0.2-scope.md](./13-v0.2-scope.md)  
> **原则**：以 **现有 cloud 数据** 为准；通过 **显式翻译表** 让列表接近 Mock/概念稿体验，**不臆造业务事实**（距离/同小区本期用 Mock 占位）。

---

## 1. 目标

在 **不改列表 UI 组件**（`TaskCard`）前提下：

1. **taskType**：由 `status` + `nodeDef.name` 经 **翻译表** 输出 Mock 风格短文案（如「联系客户」「上门服务」），不再用长 `title` 或裸节点名。  
2. **Tab**：保留 Web 四 Tab 文案（待接单 / 待联系 / 待上门 / 待跟进），用 **status→group 全量归类**；BFF 按 Tab **带 `in:status` 查询**，计数与列表一致。  
3. **地址**：**本期落地** 区县码→名称（如 `110105`→朝阳区），常规地址问题一次解决。  
4. **暂缓**：驾车距离、同小区单数 → **沿用 Mock 占位**；列表图标 → **维持当前 Mock 字符图标**，不接 CDN 图。

**范围外**：详情页、跟进记录、资料区 → v0.2.2+。

## 3. taskType：双层翻译（Mock 体验）

### 3.1 规则（优先级）

```
taskType =
  STATUS_TO_TASK_TYPE[status]           // ① 状态 → Mock 风格动作名（主）
  ?? NODE_DEF_TO_TASK_TYPE[nodeDef.name] // ② 当前节点名 → 动作名（补）
  ?? shorten(statusLabel(status))        // ③ 码表中文缩短
  ?? "处理中"
```

- **`title`**：始终保留 `data.title`（长标题），仅用于搜索/详情，**不**用于列表主标题。  
- **`statusText`**：始终为 `statusLabel(status)`（码表原文，如「待签约」），用于角标/tags。

## 4. Tab：语义归类 + BFF 查询

### 4.1 Web Tab（文案不变）

| `filter` key | UI 文案 |
| --- | --- |
| `to_accept` | 待接单 |
| `need_contact` | 待联系 |
| `onsite` | 待上门 |
| `following` | 待跟进 |
| `all` | 全部 |

### 4.2 表 — `status` → `group`（全量 23 码）

| group | 包含 status |
| --- | --- |
| **to_accept** | 101, 102, 103 |
| **need_contact** | 104 |
| **onsite** | 105, 202 |
| **following** | 200, 201, 203, 204, 205, 206, 207, 300, 301, 302, 401, 402, 403, 407, 501, 502, 999 |

说明：

- **105 待预约** 归入 **onsite**（与「待上门」同一管家任务带，避免 Mock「待上门」Tab 漏单）。  
- **200/201** 归入 following（非上门态）。  
- 小程序菜单 Tab 比 Web 更细；本期 **不改 Web Tab 个数**，只保证 **归类稳定、可查询**。

### 4.3 BFF 行为（0.2.1 必做）

| 请求 | 行为 |
| --- | --- |
| `GET /api/work-orders` 无 filter | `orderState=all`，不传 status 限制 |
| `GET /api/work-orders?filter=onsite` | 追加 `in:status\|array#and=105,202`（逗号分隔，与小程序一致） |
| 其它 filter | 按 §4.2 表传对应 status 列表 |
| Tab `count` | 对 **同一查询条件** 下的列表做 `group` 聚合；或 **并行 4 次轻量查询**（`rows=1` 仅取 total）— 实现时二选一，以 **与列表数据一致** 为准 |

不再「拉全量 orderState=all 再前端筛」作为 **唯一** 路径（避免计数与列表不一致）。

## 6. 地址：区县码表（本期交付）

### 6.1 问题

cloud 返回 `district: "110105"`（行政区划码），直接拼接会得到 `110105 日坛东一街`；小程序用 `area.js` 的 `county_list[110105]` → `朝阳区`。

### 6.2 方案

1. 自 `code/app/business/resources/js/area.js` **提取** `county_list`（及必要时 `city_list`）为静态 JSON：  
   `business_3_0/web/lib/data/area-county.json`（仅县级，体积约数十 KB 级）。  
2. 提供 `formatDistrict(code: string): string`：命中则返回县名，否则返回空（仅展示 `address`）。  
3. 可选：脚本 `scripts/extract-area-county.mjs` 在 business 仓库更新 area 后重跑，写入 xlink-app。

**验收**：列表地址以 **「朝阳区 日坛东一街」** 形式出现，不出现裸 `110105`。

## 7. 码表服务

文件建议：`lib/cloud-code-labels.ts`

- 启动或首次 cloud 读时拉取 `STATUS`、`repairParts`（实测 `codeId` 分别为 `STATUS`、`repairParts`）。  
- 内存缓存 + TTL（24h）；失败时 taskType 走表 ②，statusText 回退为 status 原值。  
- 导出：`statusLabel`、`partLabel`、`loadCodeLabels()`。

翻译表（§3.2、§4.2）为 **代码内常量**（可注释「来源：beta STATUS 全表 + 产品确认」），与动态码表分工：

- 动态码表 → `statusText`、未知 status 的兜底文案。  
- 常量表 → `taskType`、`group`（稳定、可测）。

## 9. 验收标准

1. **taskType**：beta 列表首条 **≠** `data.title` 全文；104 态为 **「联系客户」**；202 态为 **「上门服务」**（有数据时）。  
2. **Tab**：`filter=need_contact` 时仅含 status 104；`filter=onsite` 含 105/202；「全部」计数 ≥ 单 Tab 计数。  
3. **地址**：含 **区县中文名**，无裸 6 位区划码前缀。  
4. **占位**：distance / nearbyCustomers 与 Mock 表现一致（允许占位，不阻塞发版）。  
5. `npm run verify:cloud-read` **PASS**；响应头 `X-Xlink-Read-Source: cloud`。
