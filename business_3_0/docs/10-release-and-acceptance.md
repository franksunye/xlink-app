# 版本线、验收与文档治理

> 与 [06-evolution-and-poc-waves](./06-evolution-and-poc-waves.md)、[08-milestones](./08-milestones.md)、[07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md) 对齐；**可发布版本**以 `business_3_0/web/` 与本文流程为准。

## 1. 版本线含义

| 线 | 含义 | 与 POC 波次 |
| --- | --- | --- |
| **v0.1** | **POC 可演示**：架构（Next + BFF + Mock）+ IA 主路径 + [07](./07-nonfunctional-and-poc.md) 缓存/SW 基线在代码/文档可查证 | 对应 **波次 1**（[09-v0.1-scope](./09-v0.1-scope.md)） |
| **v0.2** | **POC 延伸**：在 v0.1 基线上增加与 [08-milestones](./08-milestones.md) 第 2 条一致的「节点/表单占位或只读」等 UI 壳，仍 **不**以生产承诺为目标 | 建议对应 **波次 2**（只读链路与 spec 方向见 [06](./06-evolution-and-poc-waves.md)） |
| **v0.3+** | 按 [06](./06-evolution-and-poc-waves.md) 波次 3+ 拆特性包；仍为 **Preview/内测** 向，直到团队单独定义 **Beta** | 波次 3、4… |
| **Beta（可为 v1.0-beta 等标签策略）** | **有限真实用户/内测 URL**，需稳定的回滚与已知问题清单；**不**等同现网 `business` 替代 | 跨波次集合，由发版说明约定 |
| **Prod-facing** | **对外生产承诺**的运行实例：除功能外须满足运维、安全、合规、SLO 等（超出当前 v0.x 文档范围，单独立项） | 非 POC 线；发版前另批「生产就绪」条目 |

**约定**：未特别声明时，**v0.x = POC/Preview 语义**；升级到 Beta/Prod-facing 时须在范围文档与里程碑中 **显式写明**。

## 2. 范围冻结：单一规则（真源）

**规则**：每个 **已规划发布的版本** 使用 **一份独立、冻结后的范围文件**，命名 **`NN-vX.Y-scope.md`**（`NN` 为目录内递增序号，与 [09-v0.1-scope](./09-v0.1-scope.md) 同形）。

**理由（简要）**：

- **不可变快照**：与 Git tag 一一对应，避免「同一份 09 被后续编辑」导致说不清某 tag 到底装了什么。
- **滚动更新仅用于「下一版草稿」**：下一版在 **新文件** 中撰写，直至进入「范围冻结」小节再打状态。

**不推荐**：长期只维护一份滚动 `09-*.md` 覆盖多版本——审计与对比成本高。

## 3. 生命周期（每版本）

### 3.1 范围冻结入口

| 阶段 | 动作 |
| --- | --- |
| 起草 | 新建或更新 `NN-vX.Y-scope.md` 草稿 |
| **冻结** | 将文内 **状态** 改为「范围已冻结」；**不再**增删「范围内」大项；缺陷走变更单或下一版范围文件 |
| 实现 | 仅在该版本范围内改 `web/` 与关联文档 |
| 打标前 | 执行 [§4 验收清单](#4-验收清单模板可发布版本) 与 [§5 文档更新](#5-打标-tag-前须完成的文档最小集) |

### 3.2 与里程碑行的对应

见 [08-milestones](./08-milestones.md) 及下表 **出口门禁**。

## 4. 验收清单模板（可发布版本）

以下为 **v0.x / POC 发版** 最小集；Beta/Prod 在此外另加安全、监控、回滚等条目。

| # | 项 | 通过标准 |
| --- | --- | --- |
| 1 | **构建** | `cd business_3_0/web && npm run build` 无错误退出 |
| 2 | **部署 URL（Vercel + 域名，若已接平台）** | **若**本仓库已关联 Vercel **且** 已在项目中配置 **生产 / 预览** 所用域名：存在与本次 tag/提交对应的 **有效 HTTPS 链接**（自定义域或平台默认域均可），可演示主路径；**未关联或未配域时不作为**本版验收阻塞项（与 [09-v0.1-scope](./09-v0.1-scope.md)「术语」与「域名与每次部署」两节一致） |
| 3 | **手动脚本** | 按对应 `NN-vX.Y-scope.md` 中「成功标准」与 [flows/](./flows/) 走通主路径（v0.1：`login` mock → 工作台 → 任务列表/详情 → 项目列表等） |
| 4 | **视觉验收（管家关键屏）** | **工作台、工单列表/详情、项目列表、登录** 对照 [archive 概念版 `business_3_0`](https://github.com/franksunye/xlink-app/tree/archive/business-3-0-wip/business_3_0)：**留白与间距、字体层级、区块主次**与概念版基调一致（见 [09-v0.1-scope](./09-v0.1-scope.md)「管家侧视觉 / UX 验收」；不要求与 uni-app 全屏像素逐点等同） |
| 5 | **缓存头抽查** | 对 **静态壳** / **BFF JSON** / **登录或用户私有接口** 各抽 1～2 个代表路径，核对与 [07](./07-nonfunctional-and-poc.md) 分层一致（非「全开 `public` 长缓存」） |
| 6 | **SW 与 `/api`** | **默认**规则：**不**对 `/api`（或项目约定的 BFF 前缀）做宽泛缓存；若启用 SW，须在代码或可链文档中可验证 Workbox 排除或等效策略 |

## 5. 打标（tag）前须完成的文档最小集

| 文件 / 产物 | 必须改动 |
| --- | --- |
| [08-milestones](./08-milestones.md) | 对应 **里程碑行** 或表 **版本 \| 范围 \| 验收** 中标记本版本状态（如「已释放 v0.1」） |
| `NN-vX.Y-scope.md` | **状态** 更新为「已发布」或等价措辞；日期可选 |
| [changelog.md](./changelog.md) | **一行** 日期 + 版本 + 一句概要（见该文件约定） |

**刻意不纳入最小集**：`web/CHANGELOG.md`（若未来单独建站再设）；长文 Release Note 按需链接 PR 或讨论帖。

## 6. 里程碑逐步「完成」与出口门禁

与 [08-milestones](./08-milestones.md) 表格两行及 POC 波次对齐：

| 里程碑（08 顺序） | 建议版本 | 波次 | 出口门禁（全部为真才可 bump / 标记完成） |
| --- | --- | --- | --- |
| **1** — Next.js SPA/PWA 脚手架、Mock、管家主路径壳、缓存基线 | **v0.1** | 1 | ① 对应范围文件已 **冻结并完成**（[09-v0.1-scope](./09-v0.1-scope.md)）② [§4](#4-验收清单模板可发布版本) **六项**全过 ③ [§5](#5-打标-tag-前须完成的文档最小集) 完成 |
| **2** — 节点/表单占位或只读，对照 `business` 工单语义走通 UI | **v0.2** | 2 | ① **新增** `NN-v0.2-scope.md` 且已冻结 ② 实现与 [flows/](./flows/) 已定稿部分 **无未解决阻塞**（已知局限写入范围「范围外」）③ [§4](#4-验收清单模板可发布版本)+[§5](#5-打标-tag-前须完成的文档最小集) 再执行一遍 |

**bump 版本**＝：范围冻结 + 验收清单全过 + 文档最小集落地 + Git tag（命名示例：`v0.1.0`，团队可另行 semver 细则）。

## 7. 角色与产物：真源与归档

| 类型 | 真源（Source of truth） | 归档 / 只读参照 |
| --- | --- | --- |
| **某版本装了什么** | 该版本的 **`NN-vX.Y-scope.md`** + Git **tag** 指向的代码 | 发布后不改 scope 正文；修正用 **新 patch 或下一版文件** |
| **路线与顺序** | [08-milestones](./08-milestones.md)、[06-evolution-and-poc-waves](./06-evolution-and-poc-waves.md) | 历史决策可留在 Git 历史与 [changelog.md](./changelog.md) |
| **非功能门禁** | [07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md) + 代码中可审计的头部与 SW 规则 | — |
| **流程细节** | [flows/](./flows/) | flows 变更记入 PR；与某版本绑定关系在对应 scope 中引用 |

**发布后**：对应 scope 文件视为 **该片发布的规格快照**；日常迭代在 **下一版 scope 草稿** 或主线设计文档中进行。
