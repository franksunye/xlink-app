# 与现有 cloud / cloud_ui 的集成与脱离策略

> 2026-05-14 起稿：回应「完全重建是否理想化」与「第一期接现有 cloud，后续如何脱离」。

## 目标张力

- **理想终局**：`cloud`、`cloud_ui` 与 3.0 所代表的新栈、新领域模型**完全重建**、统一在清晰边界内演进。  
- **现实约束**：一次性全量重建风险高、周期长；第一期往往需要**先跑通价值路径**（你已倾向 Mock / 再接真实环境）。

结论不必二选一极端：采用 **绞杀者模式（Strangler）** —— **3.0 自有边界清晰**，对外通过**稳定契约**与现网协作；**实现**从「接旧 cloud」逐步换成「新实现」，而不是让 3.0 业务代码直接长进旧模块里。

## 第一期接现有 cloud 时，建议的「防粘住」结构

1. **防腐层（Anti-Corruption Layer, ACL）**  
   - 在 3.0 工程内（或独立 BFF）定义 **3.0 自己的领域类型与用例接口**。  
   - 为「工单、流程节点、用户会话」等建 **Adapter**：内部调用**现有** `cloud` HTTP/RPC，把响应**映射**进 3.0 模型。  
   - **禁止**在页面与业务逻辑里散落「旧 API 字段名」；只允许 Adapter 知道旧契约。

2. **契约归 3.0 拥有（Contract-first）**  
   - 用 **OpenAPI / GraphQL schema / 事件 schema**（择一或组合）描述 **3.0 对外需要的能力**，先作为**目标契约**。  
   - 第一期由 **Adapter 实现**该契约（背后是旧 cloud）；第二期起同一契约改由 **新 cloud 服务**实现，3.0 业务代码**不动或少动**。

3. **契约测试与黄金场景**  
   - 对每条关键用例（如管家主路径）维护 **契约测试**（或基于录制的快照 + 关键断言）：Adapter 对接旧 cloud 时必须绿。  
   - 换新实现时：**同一套测试**绿即视为可切换；这是「脱离」的客观门槛。

4. **按限界上下文分批替换**  
   - 脱离不是「一天拔插头」，而是 **一块能力、一条边界** 地换：例如先换「读模型」再换「写模型」，或先换「调度」再换「结算」。  
   - 每块完成后 **删除或收窄** 对应 Adapter，避免双轨无限期并行。

## 「脱离」在工程上长什么样

| 阶段 | cloud / cloud_ui | 3.0 侧 |
| --- | --- | --- |
| 一期 | 继续为事实来源 | ACL + Mock 并存；真联调时仅 ACL 触网 |
| 中期 | 部分接口由新服务实现 | 同一契约下切换路由或网关指向新实现 |
| 后期 | 旧实现下线 | Adapter 删除或退化为只读迁移工具 |

**数据与事务**往往比 API 更难：若终局要换库或换模型，需单独规划 **迁移批次、双写/对账、回滚**；与「前端 + BFF 防腐层」解耦，可并行准备。

## 说明驱动：机器可读 spec 为唯一真源

- **流程与状态**：以 **机器可读 spec**（如状态机定义、工作流步骤 YAML/JSON、与代码共仓版本化）为真源；**不依赖**长篇叙述文档。  
- **落地方式**：spec 驱动 **类型生成、校验、测试用例或部分 UI 约束**（按团队工具链选型）；人读文档仅保留「例外与运维说明」即可。  
- 与 [02-vision](./02-vision.md) 中「业务不变」一致：spec 描述的是**对外行为与状态语义**，实现可换。

## 与「完全独立重建」的折中表述

- **心智上**：3.0 代码与模型 **自洽、独立**；**不**把旧 cloud 的领域模型当作 3.0 的内部模型。  
- **时间上**：允许 **第一期** 通过 ACL **调用**旧 cloud；**脱离** = 在**同一契约**下替换实现 + 收缩 Adapter，而非推翻 3.0 应用层。

## BFF 形态（已定稿）

- **默认采用选项 A**：BFF 与 **Next 同仓、同部署**（例如 `app/api/**/route.ts` 或项目约定路径），与 **StockWise** 前端一致——该仓库在 `frontend/src/app/api/` 下大量使用 Route Handlers 承担 BFF，迭代效率高。  
- **3.0**：沿用同一心智——浏览器只调自有 API，Route Handler 内做聚合、鉴权与 **Adapter（对接 Mock / 旧 cloud）**；若未来确有独立扩容或网关切流需求，再将部分路由迁出为独立服务（选项 B），而不必**一开始**就做独立 BFF。

## 仍待定的少量决策（可选）

- **契约仓库位置**：与 `business_3_0` 同仓 `spec/` 还是独立 `api-contracts` 仓（影响协作与 CI）。
