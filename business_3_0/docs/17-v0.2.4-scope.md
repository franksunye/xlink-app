# Web v0.2.4 — 前端技术地基（PWA 读性能 + App 壳）

> **状态**：范围已冻结（共识稿，2026-05-16）  
> **前置**：[16-v0.2.3-scope](./16-v0.2.3-scope.md)（真实登录 + 读贯通）、[07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md)（缓存/SW 从第一天）  
> **产品版本**：`business_3_0/web/package.json` → **0.2.4**

---

## 共识摘要

- **本版目标**：把「工具型 App」的**公共前端地基**做扎实——持久 Tab 壳、React Query 读缓存、PWA 网络策略、体感性能——**不是**只做底部 Tab 条样式。
- **v0.2.3 已证明**：同一 JSESSIONID 下 Dashboard / 任务 / 项目 / 我的 **读贯通**；v0.2.4 证明 **壳层切换与缓存** 达到可验收的 App 体感。
- **核心手段**：`AppTabShell` 保活已访问 Tab + TanStack Query `staleTime`/`gcTime` + `isInitialQueryLoad`（有缓存不骨架）+ SW **network-only**（不缓存 API/HTML 壳）。
- **详情与 Tab 分离**：四个 Tab 走 Shell；`/work-orders/[id]` 等仍走 App Router `children`，返回列表时任务 Tab **未卸载**。
- **明确不做**：离线写、虚拟列表、View Transitions、写路径与乐观更新（见 [范围外](#范围外-v024-及以后)）。

---

## 背景与问题（v0.2.3 体感）

| 现象 | 根因 |
|------|------|
| 切 Tab 像重新打开页面 | App Router 在 Tab 路由间 **卸载/重建** `page.tsx` |
| 每次全屏骨架 | `useQuery` + `isPending`，remount 后即使有缓存也闪 loading |
| 「我的」每次加载 | 曾用 `useEffect` 直拉；v0.2.4 已迁 **React Query** |
| 列表筛选/滚动丢失 | 组件 state + 卸载；Shell 保活 + URL `?filter=` 缓解 |

`AppTabBar` 已用 `next/link`（非硬刷新）；体感问题主要来自 **卸载 + 数据/UI 重置**。

## A. Shell 与导航（App-like）

### A.1 持久 Tab Shell

| 项 | 约定 |
|----|------|
| 布局 | `(app)/layout.tsx`：`AppTabShell` + `AppTabBar` |
| 保活 | 首次访问的 Tab 加入 `mounted` Set，`hidden` + `inert` 隐藏非激活面板 |
| 路由占位 | Tab 根 `page.tsx` 仅 `return null`，避免与 Shell **双渲染** |
| 实现 | `components/AppTabShell.tsx`、`lib/tab-navigation.ts`、`components/screens/*` |

**验收**

- [x] 四个 Tab URL 仍为 `/`、`/work-orders`、`/projects`、`/account`（可分享/刷新）
- [x] 切换 Tab 不卸载已访问过的 `*Screen` 组件
- [ ] 冻结后人工：连续切换四 Tab **无整页白屏/路由级 loading**

**状态**：**已完成（核心）**

### A.2 每 Tab 滚动保留

| 方案 | 说明 |
|------|------|
| **推荐（v0.2.4）** | 依赖 Shell **DOM 保活**：各 Tab 面板 `display: hidden` 时滚动容器仍在文档流外保留 `scrollTop` |
| TabBar | `Link` 已设 `scroll={false}`，避免 Next 重置窗口滚动 |
| 列表容器 | 任务/项目列表使用面板内 `overflow-y-auto`（或自然文档滚动），**勿**在 Tab 切换时 `key` 重置列表 |
| 加强（P1，非阻塞） | 若某屏 remount 仍丢滚动：对该 Tab 根节点 `useRef` 记录 `scrollTop`，`useEffect` 在 `active` 时恢复 |

**验收**

- [ ] 任务列表滚至中部 → 切「我的」→ 回任务：**滚动位置大致保留**（±1 屏容差）
- [ ] 工单 `?filter=` Tab 筛选在 Shell 保活期间保留

**状态**：**依赖保活，未单独加 scroll 恢复代码** → 见 [Implementation backlog](#implementation-backlog)

### A.3 详情栈 vs Tab 路由

| 类型 | 行为 |
|------|------|
| Tab 根 | `tabIdFromPathname` 命中 → 仅 Shell 渲染对应 `*Screen` |
| 非 Tab（如 `/work-orders/[id]`） | `AppTabShell` 返回 `children`；底部 Tab **任务高亮**（`pathname.startsWith('/work-orders/')`） |
| 返回列表 | 浏览器后退或列表内 `Link` → 任务 Tab 面板仍在 `mounted` 中 |

**验收**

- [x] `tab-navigation` 单测：详情 path 返回 `null`
- [ ] 人工：进详情 → 返回 → 列表无全屏骨架（RQ 有缓存时）

**状态**：**路由模型已完成**；详情页 loading 条件见 backlog（`isInitialQueryLoad`）

### A.4 Prefetch 策略

| 层级 | 策略 |
|------|------|
| RSC / 路由 | `AppTabBar`：`prefetch` + `scroll={false}`（**已做**） |
| 数据（可选 P1） | 登录成功后 `queryClient.prefetchQuery` 预热 `dashboard`、`work-orders`、`projects`、`/api/auth/me` |
| Shell mount（可选 P2） | `AppTabShell` 首次 mount 对未访问 Tab **不**强制挂载（避免首屏四路请求）；仅 prefetch 路由 payload |

**验收**

- [x] Tab `Link` 带 `prefetch`
- [ ] （可选）登录后 2s 内切 Tab，首屏 skeleton 时间明显短于冷启动

## B. 数据与渲染

### B.1 React Query 全局默认（`AppProviders`）

| 选项 | 值 | 理由 |
|------|-----|------|
| `staleTime` | `60_000` | Tab 切换 60s 内不重复打满 API |
| `gcTime` | `10 * 60_000` | 离开 Tab 10min 内仍可瞬时展示 |
| `refetchOnMount` | `false` | 与 Shell 保活配合，避免 remount 风暴 |
| `refetchOnWindowFocus` | `false` | 弱网/内测减少意外刷新 |
| `retry` | `2` | 短抖动可恢复 |

**验收**

- [x] `AppProviders.tsx` 与上表一致
- [ ] DevTools Network：60s 内 Tab 来回 **无重复** `GET /api/dashboard` 等（允许后台 `isFetching` 静默刷新若日后开启）

### B.2 无缓存命中时的全屏骨架

| 项 | 约定 |
|----|------|
| 工具函数 | `lib/query-display.ts` → `isInitialQueryLoad(isPending, data)` === `isPending && data === undefined` |
| Tab 四屏 | `HomeScreen`、`WorkOrdersScreen`、`ProjectsScreen`、`AccountScreen` 使用上述条件 |
| 详情 | **应**同样使用（当前 `[id]/page.tsx` 仍用 `isPending` → P1） |
| placeholder | v0.2.4 **不强制** `placeholderData`；有 `data` 即展示，后台 refetch 可后续加 subtle 指示 |

**验收**

- [x] 四 Tab 二次进入无「仅因 isPending」的全屏骨架
- [ ] 详情二次进入（同 id）同上

### B.3 全 Tab 读路径 on React Query

| 屏 | queryKey | API |
|----|----------|-----|
| 首页 | `["/api/dashboard"]` | `GET /api/dashboard` |
| 任务 | `["/api/work-orders", filter]` | `GET /api/work-orders` |
| 项目 | `["/api/projects"]` | `GET /api/projects` |
| 我的 | `["/api/auth/me"]` | `GET /api/auth/me` |

**验收**

- [x] 四屏均为 `useQuery` + `fetchJson`
- [x] Account 不再 `useEffect` 直拉用户

### B.4 登录后 prefetch（可选）

```ts

await Promise.all([
  qc.prefetchQuery({ queryKey: ["/api/dashboard"], queryFn: ... }),
```

**验收**：登录后首次点「任务」**无**全屏骨架（P1）

## C. PWA / 网络

### C.1 Service Worker 策略

| 项 | 约定 |
|----|------|
| 文件 | `public/sw.js` |
| 策略 | **network-only**：`fetch(event.request)` 直转发，**无** precache / runtime cache |
| `/api/*` | 显式 `fetch(..., { cache: "no-store" })` |
| 注册 | `ServiceWorkerRegister`：**仅 production** |
| 原因 | 避免 SW 缓存 HTML/BFF 导致 **陈旧会话、难排查的 401/脏数据**；与 [07](./07-nonfunctional-and-poc.md)「SW 只做补充、API 默认不缓存」一致 |

**验收**

- [x] `sw.js` 无 Workbox precache 列表
- [x] `/api/*` 走 `no-store`
- [ ] 验收脚本：离线时 **不**展示过期私有列表（应失败态或网络错误，而非假数据）

### C.2 Manifest 与安装

| 项 | 约定 |
|----|------|
| 来源 | `app/manifest.ts` |
| `display` | `standalone` |
| `start_url` | `/` |
| `theme_color` / `background_color` | 与 `--xlink-primary` / 页面底一致 |
| icons | `/icon` 192；**P1** 补 512 + `maskable` 以利安装提示 |

**验收**

- [x] `GET /manifest.webmanifest` 200
- [ ] Chrome Lighthouse PWA：**installable** 可选通过（非阻塞）

### C.3 禁止缓存项

| 路径 | HTTP / 客户端 |
|------|----------------|
| `/api/*`（含 auth） | BFF：`Cache-Control: private, no-store, must-revalidate`（`lib/http.ts`、`middleware`） |
| Cloud 出站 | `cache: "no-store"` |
| Cookie | `xlink_b3_sess`、`xlink_b3_profile` — **永不**进 SW Cache API |
| 静态 | Next 自带 hashed assets；**不**由 SW 二次缓存 |

### C.4 安装性检查清单（可选）

1. HTTPS 部署  
2. manifest + SW 注册  
3. 192/512 icons  
4. `display: standalone`  
5. 用户手势「添加到主屏幕」后启动 URL 为 `/`
