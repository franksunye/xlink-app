# 第 1 轮：定位与边界

> 状态：已定稿（2026-05-14，对话收敛；细节随多期迭代再补）

## 一句话目标

用**新的技术栈**搭建可跑通的**整体系统架构**（工程与分层清晰），并在多期迭代中持续交付；首要「变好」的是**架构与工程效率**，而非先改业务故事本身。

## 不变量（与现网必须一致）

- **对用户的业务含义**：已发生的业务事实是约束；面向师傅/业主等的**业务能力与规则**不因 3.0 而另起一套「平行宇宙」。
- **前端能力对齐参照**：移动端以现网 **`business/`** 为功能对齐基准；管理端以 **`cloud_ui`** 为功能对齐基准。
- **后端能力对齐参照**：以后端 **`cloud`** 为能力对齐基准；迭代版本将重新定义，但需有**版本化/迁移策略**，避免与移动端、管理端长期脱节。
- **后端领域模型**：允许在 **cloud** 侧做**规范、优化、收敛**（既往建模与工程不足需补课）；须与「对外业务行为一致」协调（见备注）。

## 明确不做（本轮或本期范围外）

- **不设固定「不做功能清单」**：除**具体业务事实与规则**外，技术栈、架构、后端建模等均可纳入演进范围，由多期版本切分。
- **早期版本**：**不要求**与真实 API 联调；优先基于**新架构**的高效率开发方式推进（例如契约先行、Mock、本地/仿真环境等，具体在后续轮次选定）。

## 第一期环境（可选）

- **默认对照环境**：多期迭代，早期以**架构跑通与工程效率**为主；与真实环境对齐的时机放到后续版本定义中（待定具体 dev/beta/prod 节奏）。
- **是否同一套 API 联调**：早期 **否**（无需真实 API）；与 `business` / `cloud_ui` / `cloud` 行为对齐在适当时机通过联调或对照测试落实。

## 备注

- **张力说明**：「业务不变」与「后端领域模型收敛」并存时，建议用**对外契约（API/事件/权限语义）**冻结用户可见行为，内部模型重构在版本边界内完成；否则易出现「以为业务没变但实际接口语义漂移」。
- **下一轮建议**：第 2 轮补**角色与典型场景**（仍可对齐 `business` 真实用法），便于第 3 轮 IA 与第 4 轮流程与参照端一一映射。
