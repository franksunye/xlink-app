# 第 1 轮：定位与边界

> 状态：已定稿（2026-05-14，对话收敛；细节随多期迭代再补）

## 一句话目标

用**新的技术栈**搭建可跑通的**整体系统架构**（工程与分层清晰），并在多期迭代中持续交付；首要「变好」的是**架构与工程效率**，而非先改业务故事本身。

**交付面（补充）**：3.0 主工程采用 **Next.js + Vercel** 的 **Web** 形态推进迭代；**早期各版本统一按单页应用（SPA）方式组织**，并落实为 **PWA**（安装与离线等能力按迭代逐步打开）。**不以微信小程序作为主迭代面**（避免小程序发版节奏拖累实验）。体验与信息架构的「第一步样子」可对齐分支 **`archive/business-3-0-wip`** 中的 [`business_3_0`](https://github.com/franksunye/xlink-app/tree/archive/business-3-0-wip/business_3_0) 概念实现，技术栈另起，不在该 uni-app 骨架上延续。

## 不变量（与现网必须一致）

- **对用户的业务含义**：已发生的业务事实是约束；面向师傅/业主等的**业务能力与规则**不因 3.0 而另起一套「平行宇宙」。
- **前端能力对齐参照**：移动端以现网 **`business/`** 为功能对齐基准；管理端以 **`cloud_ui`** 为功能对齐基准。
- **后端能力对齐参照**：以后端 **`cloud`** 为能力对齐基准；迭代版本将重新定义，但需有**版本化/迁移策略**，避免与移动端、管理端长期脱节。
- **后端领域模型**：允许在 **cloud** 侧做**规范、优化、收敛**（既往建模与工程不足需补课）；须与「对外业务行为一致」协调（见备注）。

## 明确不做（本轮或本期范围外）

- **不设固定「不做功能清单」**：除**具体业务事实与规则**外，技术栈、架构、后端建模等均可纳入演进范围，由多期版本切分。
- **早期版本**：**不要求**与真实 API 联调；优先基于**新架构**的高效率开发方式推进（例如契约先行、Mock、本地/仿真环境等，具体在后续轮次选定）。**应用壳为 SPA 式 PWA**，与「先跑通架构与主路径」一致。

## 第一期环境（可选）

- **默认对照环境**：多期迭代，早期以**架构跑通与工程效率**为主；与真实环境对齐的时机放到后续版本定义中（待定具体 dev/beta/prod 节奏）。
- **是否同一套 API 联调**：早期 **否**（无需真实 API）；与 `business` / `cloud_ui` / `cloud` 行为对齐在适当时机通过联调或对照测试落实。

## 备注

- **张力说明**：「业务不变」与「后端领域模型收敛」并存时，建议用**对外契约（API/事件/权限语义）**冻结用户可见行为，内部模型重构在版本边界内完成；否则易出现「以为业务没变但实际接口语义漂移」。
- **与现网 cloud 分期关系**：见 [integration-strategy.md](./integration-strategy.md)（第一期可接旧 cloud + 防腐层，终局在自有契约下替换实现）。
- **说明驱动**：以**机器可读 spec** 为真源（流程/状态机与代码共仓），少依赖叙述文档；详见 integration-strategy 第四节。
- **下一轮建议**：第 3 轮 **IA** 可同时对照 **`business/`**（业务语义）与 **`archive/business-3-0-wip` 的 `business_3_0`**（Web 化时要保留/改良的导航与页面拆分），输出 Next.js 下的路由与布局草案。
