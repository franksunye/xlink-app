## 缓存与 PWA（从第一天起必选）

**背景（团队内教训）**：同类项目 **StockWise**（Next + Route Handlers + PWA）未在初期系统性设计缓存，后续为性能与一致性付出**大量重构与妥协**。3.0 **避免重复该路径**。

启用 PWA 后，常见「服务器与性能」压力来自：

- **Service Worker**：若对 HTML/API 缓存过宽或失效策略不清，会导致**陈旧数据**、**回源风暴**或**难以排查的线上行为**；离线队列与重放还会改变**写路径**假设。  
- **BFF（选项 A）**：浏览器与 SW 的请求最终打到 **Next 的 `route.ts`**；若无显式 **`Cache-Control` / `CDN-Cache-Control`** 与客户端去重策略，**边缘与源站**易被放大。  
- **SPA 数据层**：列表/详情重复拉取、无 `staleTime` 的「每次 mount 都打满」，在弱网下体感差、对源站也不友好。

**从第一天建议落实的约定（可随脚手架写进模板）**：

1. **分层**：至少区分 **静态壳**（可长期或版本化缓存）、**BFF JSON**（短缓存或不可缓存 + ETag）、**用户私有数据**（默认 `private, no-store` 或极短 `max-age`）。  
2. **HTTP 为先**：优先用 **响应头** 表达可缓存性；SW 只做 **补充**（例如壳的 precache、`navigate` 的 network-first 等），且**文档化**每条路由属于哪一类。  
3. **客户端数据**：引入一种稳定的数据获取范式（如 SWR / TanStack Query），为列表与详情约定 **`staleTime`、去重、后台刷新`**，与 PWA 并存时仍可控。  
4. **SW 上线节奏**：可先 **仅 manifest + 不缓存 API** 的 SW，或 Workbox 规则显式 **排除 `/api/*`**，再按模块收紧；**禁止**默认「缓存一切」直到策略评审通过。

## 可观测与发布

> 本节**后续完善**；设计阶段先对齐业务侧已有概念。

- **业务可观测（参照现网）**  
  - **Activity（活动/操作轨迹）**：用户或系统关键操作的时间线类记录，用于还原「谁做了什么」。  
  - **Audit log（审计日志）**：合规与追责维度，字段级或接口级变更、敏感操作留痕。  
- **工程可观测**：日志级别、请求 ID、BFF 错误上报、与 Vercel Analytics / 外部 APM 的集成 —— 在波次 1 脚手架中预留钩子即可，细则迭代补。  
- **埋点**：与现网埋点命名对齐策略待定；不阻塞首期 Mock 竖切。  
- **发布与并行期**：与 `business/` 并行时默认用户侧仍以现网为准；3.0 Web 为实验/内测 URL；Vercel Preview 策略随脚手架补细则。

## 备注

- **Next.js 与 SPA**：早期以 **客户端路由 + 客户端数据获取** 为主即可满足 SPA 形态；若某屏需 SEO 或首屏极致优化，再单独引入 SSR/SSG，不作为早期默认。
- 若后续需要 **小程序壳** 仅做容器嵌入 Web（Hybrid），可作为独立 POC，不改变「主迭代在 Web」的默认。
- **StockWise 对照**：同栈下「晚做缓存设计」的成本已在团队共识中；3.0 以本节为**门禁**，脚手架评审须过。
