### 术语：Vercel Preview（预览部署）

| 用语 | 含义（给非运维同事） |
| --- | --- |
| **Vercel Preview** | 将仓库 **接入 Vercel** 之后，每一次 **推送代码** 或为仓库开的 **Pull Request**，平台会为该次构建生成 **一个临时的 HTTPS 链接**，用在 **不触碰生产域名**的情况下演示「这一版长什么样」。链接随部署生命周期存在，不等同正式上线。 |

### 域名与「每次部署都能体验」

团队将在 **Vercel 项目里配置自有域名**（生产与/或 **Preview** 子域，例如通配符预览域），使 **每次部署** 都有稳定、可分享的 **HTTPS 入口**，便于产品/业务直接打开体验，而不依赖本机 `localhost`。具体主机名以 Vercel **Domains** 与项目 **Git 分支 / 环境** 绑定为准；默认 `*.vercel.app` 与自定义域可并存。

> **门禁说明**：在仓库已接 Vercel 的前提下，**成功标准**包含：存在与本次提交/环境对应的 **可访问部署 URL**（自定义域或平台默认域均可），且能走通与本地一致的 **主路径骨架**。**仓库尚未接通 Vercel 时**，仍以本地 + `npm run build` 验收，不阻塞文档收口；域名配置就绪后应把 **生产 / 预览基 URL** 记入 `web/README.md` 或团队 Wiki（勿将敏感凭据写入 Git）。

## 目标

在 `business_3_0/web/` 中以 **Next.js + BFF（Route Handlers）+ Mock 数据** 竖切 **POC 波次 1**：证明 SPA 形态下的 **信息架构主路径**（登录 → 工作台 → 任务列表/详情、项目列表）可 **本地**演示，并在 **仓库已接通 Vercel 且已配置可访问域名/预览域** 时，用线上部署 **同等**骨架演示；同步落实 **HTTP 缓存分层** 与 **SW 对 API 的默认安全策略**（或等效文档化约定），避免「默认可缓存一切」。业务语义仍以现网 `business/` 为对照方向，v0.1 **不**承担真实后端联调与完备工作流。

### 管家侧视觉 / UX 验收（v0.1）

- **工程侧**：仍以 **Mock-first、BFF 契约稳定**为第一优先，不写死与现网 `business` **功能逐项对齐**。  
- **验收侧（面向管家体感）**：**工作台、任务列表/详情、项目列表、登录主路径** 应对齐 [archive 概念版 `business_3_0`](https://github.com/franksunye/xlink-app/tree/archive/business-3-0-wip/business_3_0) 已形成的高完成度观感（团队在文档中称约 **「80 分」 tuned**）：**留白与间距、字体层级、关键区块层次与主次**须在上述屏中与概念版基调一致；**不是**要求在像素上与 uni-app 逐屏逐组件绝对一致，也不覆盖商用后的全状态与长尾屏。详见 [10-release-and-acceptance.md](./10-release-and-acceptance.md) 验收清单中的视觉条目。

## 范围内（v0.1）

### 路由与页面（IA 主路径）

| 路由 | 说明 |
| --- | --- |
| `/login` | 登录栈占位：可多步 UI 简化，但须能 **mock 成功后进入** 受保护主区（见 [flows/login.md](./flows/login.md)）。 |
| `/` | 工作台：对齐 `getDashboard` 聚合块与跳转类型（指标、今日任务、快捷入口；见 [flows/work-orders.md](./flows/work-orders.md)）。 |
| `/work-orders`、`/work-orders/[id]` | 任务列表（含 mock `group`/`filter` 概念）与详情（摘要、`context`、`followRecords`、`timeline` 等块壳）。 |
| `/projects` | 项目列表卡片（`getProjects`）；详情 `/projects/[id]` **可选**（列表 + toast/占位即可，见 [flows/projects.md](./flows/projects.md)）。 |
| `/account` | 「我的」Tab **壳**：导航占位即可，子功能不展开。 |

全局布局：**底部或侧栏 Tab** 与 [04-information-architecture](./04-information-architecture.md) 一致（首页 / 任务 / 项目 / 我的）；命名建议第三项统一为「项目」。

### BFF Mock 端点（`GET` 为主 + 登录桩）

| 端点 | 用途 |
| --- | --- |
| `GET …/dashboard`（或等价路径） | Mock `getDashboard` 结构。 |
| `GET …/work-orders`、`GET …/work-orders/[id]` | Mock 列表与详情。 |
| `GET …/projects` | Mock `projects` 列表。 |
| `POST /api/auth/login`（或项目约定的 auth 路径） | **登录桩**：返回 mock 用户上下文（如 `userId`、`displayName`、`role`），默认 **不可被 SW 缓存**（见 [flows/login.md](./flows/login.md)）。 |

具体 URL 前缀以 `web/` 实现为准；须与前端数据层约定一致。

### 缓存与 PWA 基线

- **BFF 响应**：按 [07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md) 分类设置 `Cache-Control` / 必要时 `CDN-Cache-Control`（例如静态壳可版本化；JSON **短缓存或 `private`/`no-store`** + ETag 等策略择一明确）。  
- **客户端数据层**：列表/详情具备 **去重、`staleTime`、失败重试** 的范式（如 SWR / TanStack Query），与缓存头一致。  
- **Service Worker**：**默认不缓存 `/api/*`**（Workbox 排除或等价规则），或在 `docs` + 代码注释中 **逐条写明** 例外与失效策略；禁止未评审的「通配缓存 API」。  
- **PWA**：`manifest` **可选用** 占位；SW 可在「排除 API」前提下逐步启用。

## 范围外（v0.1 明确不做）

- **真实 cloud** 联调、**cloud_ui** 管理端迁入或复刻。  
- **支付 / 签约 / 电话** 等外部集成（仅占位文案或删除入口，不接沙箱）。  
- **完整工作流节点**、表单引擎、低代码运行时；详情页内 **主行动** 以 mock/占位为主，不接全状态机。  
- **机器可读 spec / 状态机文件**：若体积与维护成本大，v0.1 **仅允许** TypeScript 类型、常量表等 **轻量桩**；完整 spec 真源与波次 2 对齐 [05-integration-strategy](./05-integration-strategy.md)。  
- **售后工单** 等独立链路（见 [flows/work-orders.md](./flows/work-orders.md) 备注）。  
- **`/feedback`、`/security-alert` 路由与完整体验**：优先级 **极低**，**明确不在 v0.1 范围**；延至产品进入 **小规模商用就绪** 阶段再评估（与 [04-information-architecture](./04-information-architecture.md) 中标注一致）。IA 中若保留快捷入口文案，**不实现跳转或仅 toast** 即可。

## 成功标准

- `cd business_3_0/web && npm run build` **通过**。  
- **本地 `npm run dev`** 可走完：登录桩 → 工作台 → 任务列表/详情 → 项目列表（及 `/account` 壳），与 [04-information-architecture](./04-information-architecture.md) 主路径一致。  
- **文档可链接**：本文件与 [08-milestones](./08-milestones.md)、[06-evolution-and-poc-waves](./06-evolution-and-poc-waves.md) 相互引用；缓存/SW 策略在代码或 README 中可指向 [07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md)。  
- **线上部署（Vercel + 域名）**：**若** `web` 已在 Vercel 与 Git 关联 **且** 已配置用于 **Production / Preview** 的域名，则应存在可访问的部署 URL，且能走通与本地相同的 **主路径骨架**；**若尚未关联或未配域**，本条 **不作为** v0.1 文档收口或本地验收的阻塞项（见上文「术语」与「域名」两节）。

## 非目标与风险（一句话）

| 类型 | 说明 |
| --- | --- |
| 非目标（工程边界） | v0.1 **验证架构与 IA**；数据与接口为 Mock，**不**承担与现网 `business` **功能逐项对等**；亦将 `/feedback`、`/security-alert` 整体验 **排除在 v0.1 之外**（延至商用准备阶段）。 |
| 非目标（相对概念版） | **不**要求与 archive uni-app 在 **每个组件、每个状态** 上像素级逐点一致；但 **管家主路径关键屏** 的 **版式与观感**须达到上文「管家侧视觉 / UX 验收」所列 archive 基线，避免「工程 POC 级粗糙 UI」通过验收。 |
| 非目标 | 不承诺生产级安全、审计、埋点完备；仅预留钩子即可（见 [07-nonfunctional-and-poc](./07-nonfunctional-and-poc.md)）。 |
| 风险 | **SW 误缓存 API** → 陈旧数据与疑难线上行为；须代码评审门禁。 |
| 风险 | **缓存头缺失或「全开 public」** → 边缘/源站放大与隐私泄露；须按分层表落实。 |
| 风险 | **Mock 字段与现网码表脱节** → UI 自创第二套状态；须在 BFF/映射层约束命名（见 flows 中「不得 UI 自创状态机」方向）。 |
